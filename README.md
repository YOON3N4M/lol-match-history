# lol-statistics-typescript

Riot API를 활용해 **리그 오브 레전드 소환사 전적과 통계 정보를 조회할 수 있는 웹 서비스**입니다.

최근 20게임의 전적, 모스트 챔피언, 포지션 선호도, 승률, KDA 등의 통계 정보를 제공합니다.  
전체적인 디자인과 구성은 [OP.GG](https://www.op.gg/)를 참고하여 제작했습니다.

---

## 주요 기능

- 소환사 정보 조회
- 최근 20게임 전적 조회
- 게임별 상세 정보 조회
- 모스트 챔피언 7개 통계 제공
- 포지션 선호도 분석
- 승률, KDA, CS, 킬 관여율 등 주요 지표 제공
- 전적 갱신 기능
- Firestore 기반 전적 캐싱

## 프로젝트 배경

이 프로젝트는 처음에는 **React와 REST API 기반 데이터 연동을 학습하기 위해 개발한 사이드 프로젝트**였습니다.

초기 개발 당시에는 컴포넌트 구조, 데이터 흐름, 비동기 처리, 상태 관리에 대한 이해가 충분하지 않아 유지보수성과 확장성 측면에서 여러 한계가 있었습니다.

이후 프론트엔드 개발 경험을 쌓은 뒤 기존 코드를 다시 분석했고, 다음과 같은 문제를 발견했습니다.

- Riot API Key가 클라이언트에 노출될 수 있는 구조
- 클라이언트 중심의 데이터 패칭 구조
- 외부 API 요청 수 증가
- UI 컴포넌트와 도메인 로직의 결합
- API 응답 타입과 UI 가공 타입이 뒤섞인 구조

이를 개선하기 위해 기존 구조를 **Next.js App Router와 Firebase Admin 기반 서버 캐싱 구조**로 리팩토링했습니다.

---

## 주요 문제와 개선 과정

### 1. Riot API Key 노출 위험 개선

#### 문제

초기 구조에서는 Riot API 호출이 클라이언트 중심으로 이루어져, API Key와 같은 민감 정보가 노출될 위험이 있었습니다.

#### 개선

- Next.js Pages Router 구조에서 App Router 구조로 마이그레이션
- Riot API Key를 서버 전용 환경 변수로 분리
- Riot API 호출을 서버 컴포넌트, Route Handler, 서버 서비스 내부로 제한
- Firebase Admin 초기화 코드를 `server-only`로 보호
- 초기 데이터 조회 책임을 서버로 이동

#### 결과

API Key가 클라이언트 번들에 포함되지 않도록 구조를 개선했고, 데이터 요청 흐름을 서버 중심으로 단순화했습니다.

---

### 2. 서버 컴포넌트 렌더링 지연 개선

#### 문제

Riot API를 통해 한 유저의 전적을 조회하기 위해서는 여러 단계의 요청이 필요합니다.

```txt
Account 조회
→ Summoner 조회
→ League 조회
→ Match ID 조회
→ Match 상세 정보 조회
```

특히 Match 상세 정보는 최대 20개의 경기 데이터를 개별적으로 요청해야 합니다.

모든 API 응답을 기다린 뒤 렌더링을 시작하면 페이지 전환과 초기 렌더링이 크게 지연되는 문제가 있었습니다.

#### 개선

초기 화면에 필요한 데이터와 응답 시간이 오래 걸리는 데이터를 분리했습니다.

초기 화면 구성에 필요한 다음 정보는 서버 컴포넌트에서 우선 조회했습니다.

- Account
- Summoner
- League

상대적으로 응답 시간이 긴 Match 데이터 영역은 `Suspense`와 `loading fallback`을 적용해 비동기적으로 분리했습니다.

#### 결과

사용자는 기본 소환사 정보를 먼저 확인할 수 있고, 경기 기록 영역은 별도의 로딩 UI를 통해 점진적으로 표시되도록 개선했습니다.

이를 통해 모든 데이터를 기다린 뒤 한 번에 렌더링하는 구조보다 초기 대기 경험을 개선할 수 있었습니다.

---

### 3. 외부 API 사용량 절감을 위한 Firestore 캐싱

#### 문제

동일 유저를 다시 조회할 때마다 Account, Summoner, League, Match Detail 데이터를 Riot API로 직접 요청하면 외부 API 호출 수가 불필요하게 증가하는 문제가 있었습니다.

특히 Match Detail은 최대 20건까지 개별 요청이 발생할 수 있어 반복 조회 시 API 사용량이 크게 늘어날 수 있었습니다.

#### 개선

Firestore에 전적 데이터를 캐싱하고, 동일 유저 재조회 시 캐시 데이터를 우선 사용하도록 구조를 개선했습니다.

```txt
유저 전적 조회 페이지 접속
→ Firestore 캐시 데이터 조회
  → 캐시 데이터 있음: 캐시 기반 전적 UI 제공
  → 캐시 데이터 없음: Riot API 호출 → Firestore 저장 → 전적 UI 제공
```

사용자가 `전적 갱신`을 클릭한 경우에만 캐시를 무시하고 Riot API를 다시 호출하도록 구성했습니다.

#### 결과

Match Detail 최대 20건 요청을 캐시로 대체하여, 반복 조회 시 외부 API 호출 수를 줄일 수 있었습니다.

---

### 4. UI 컴포넌트와 도메인 로직 분리

#### 문제

초기 구조에서는 UI 컴포넌트 내부에 Riot API 관련 도메인 로직과 계산 로직이 함께 섞여 있었습니다.

이로 인해 컴포넌트의 책임이 명확하지 않았고, 유지보수와 확장이 어려운 구조였습니다.

#### 개선

역할에 따라 코드를 분리했습니다.

| 역할 | 경로 |
| --- | --- |
| 공통 시각화 컴포넌트 | `components/charts` |
| Riot 관련 상수 | `constants/riot` |
| KDA, 킬 관여율, 피해량 관여율, CS 계산 | `utils/riot` |
| Riot API 서비스 | `service/riot` |
| Firebase Admin 서비스 | `service/firebase-admin` |

#### 결과

UI 컴포넌트는 화면 표현에 집중하고, 도메인 계산과 외부 API 호출 로직은 별도 모듈에서 관리하도록 개선했습니다.

이를 통해 각 코드의 역할이 명확해지고 유지보수성이 향상되었습니다.

---

### 5. DTO와 UI 타입 분리

#### 문제

API 응답값의 DTO 타입과 실제 UI에서 사용하는 가공 타입이 명확히 구분되지 않아 타입의 역할이 모호했습니다.

Riot API 응답 구조와 UI 렌더링에 필요한 데이터 구조가 뒤섞이면서, 데이터 변경에 따른 영향 범위를 파악하기 어려웠습니다.

#### 개선

타입을 목적에 따라 분리했습니다.

- Riot API 응답 타입
- UI 렌더링에 필요한 파생 타입
- 계산 결과 타입

#### 결과

API 응답 구조가 변경되더라도 UI 타입에 미치는 영향을 줄일 수 있었고, 각 타입의 역할을 명확히 구분할 수 있게 되었습니다.

## 프로젝트를 통해 배운 점

이 프로젝트를 개선하며 단순히 API 데이터를 화면에 출력하는 것과, 실제 서비스처럼 안정적으로 데이터를 요청하고 캐싱하며 사용자에게 점진적으로 보여주는 구조는 다르다는 점을 경험했습니다.

또한 Server Component, Route Handler, Suspense, Firebase Admin을 함께 사용하면서 Next.js App Router 환경에서 서버와 클라이언트의 책임을 분리하는 기준을 정리할 수 있었습니다.

기능 구현뿐 아니라 API 보안, 데이터 패칭 전략, 캐싱, 렌더링 지연 개선, 타입 분리까지 함께 고려해야 실제 서비스에 가까운 구조를 만들 수 있다는 점을 배웠습니다.
